# TaskCo

TaskCo is a project and task management SaaS for small teams. It lets users manage projects, track tasks on a Kanban board, log attendance (clock in/out), and view project activity. TaskCo is **not** a multi-tenant platform and does not support real-time collaboration, file uploads, OAuth/social login, or mobile apps in v1.

## Getting Started

```bash
# 1. Clone the repository
git clone <repo-url>
cd taskco-aman

# 2. Install dependencies
npm install

# 3. Set up environment variables
cp .env.example .env.local
# Edit .env.local and fill in your Supabase project URL, anon key, service role key, and Resend keys

# 4. Push the database schema (runs all migrations in order)
npx supabase db push
# Or if using a cloud project without the CLI: paste
# supabase/migrations/COMBINED_RUN_IN_DASHBOARD.sql into the Supabase SQL editor

# 5. Start the development server
npm run dev
```

Open [http://localhost:3000](http://localhost:3000). You will be redirected to `/login` if not authenticated.

## API Routes

All routes under `/api/` are protected by `withAuth()` except `POST /api/auth/logout`.

### Auth

| Method | Path | Description |
|---|---|---|
| `GET` | `/api/auth/me` | Returns the authenticated user's id, email, full_name, and avatar_url |
| `POST` | `/api/auth/logout` | Signs out the current session (clears HttpOnly cookies); **public** |

### Projects

| Method | Path | Description |
|---|---|---|
| `GET` | `/api/projects` | List all projects (team-readable); includes embedded owner profile |
| `POST` | `/api/projects` | Create a new project; `owner_id` set server-side |
| `GET` | `/api/projects/:id` | Get a single project by ID |
| `PATCH` | `/api/projects/:id` | Update project fields; owner only (enforced by RLS) |
| `DELETE` | `/api/projects/:id` | Delete a project; owner only (enforced by RLS) |
| `GET` | `/api/projects/:id/activity` | Get the activity log for a project (last 50 entries) |

### Tasks

| Method | Path | Description |
|---|---|---|
| `GET` | `/api/tasks?project_id=<uuid>` | List tasks for a project; includes assignees and checklist counts |
| `POST` | `/api/tasks` | Create a task; `created_by` set server-side |
| `GET` | `/api/tasks/:id` | Get a single task with full details |
| `PATCH` | `/api/tasks/:id` | Update task fields; creator or assignee only (RLS) |
| `DELETE` | `/api/tasks/:id` | Delete a task; creator only (RLS) |

### Task Assignees

| Method | Path | Description |
|---|---|---|
| `GET` | `/api/tasks/:id/assignees` | List all assignees for a task |
| `POST` | `/api/tasks/:id/assignees` | Assign a user to a task; `assigned_by` set server-side |
| `DELETE` | `/api/tasks/:id/assignees?user_id=<uuid>` | Remove an assignee (by assigner or the assignee themselves) |

### Task Checklist

| Method | Path | Description |
|---|---|---|
| `GET` | `/api/tasks/:id/checklist` | List checklist items ordered by position then created_at |
| `POST` | `/api/tasks/:id/checklist` | Add a checklist item; task creator or assignee only |
| `PATCH` | `/api/tasks/:id/checklist/:itemId` | Update content, is_done, or position |
| `DELETE` | `/api/tasks/:id/checklist/:itemId` | Remove a checklist item |

### Attendance

| Method | Path | Description |
|---|---|---|
| `POST` | `/api/attendance/clock-in` | Start a work session; fails if a session is already open |
| `POST` | `/api/attendance/clock-out` | Close the current open session |
| `GET` | `/api/attendance/today` | Get today's sessions, the open session (if any), and total closed seconds |

### Profile

| Method | Path | Description |
|---|---|---|
| `GET` | `/api/profile` | Get the current user's profile (id, full_name, avatar_url, email, created_at) |
| `PATCH` | `/api/profile` | Update full_name and/or avatar_url |

## Database

Schema is managed via Supabase migrations in `supabase/migrations/`. RLS is enabled on every table.

| File | Phase | Contents |
|---|---|---|
| `001_enums.sql` | Enums | `urgency_level`, `project_status`, `task_status`, `notification_type` |
| `002_tables.sql` | Tables | profiles, projects, tasks, task_assignees, task_checklist_items, task_links, task_time_entries, attendance_sessions, calendar_events, notifications, activity_log |
| `003_functions_triggers.sql` | Functions & Triggers | `handle_new_user`, `set_updated_at`, `is_task_collaborator`, `notify_task_assigned`, `log_activity` |
| `004_rls.sql` | Row Level Security | RLS policies for every table |
| `005_views_rpc.sql` | Views & RPC | `v_task_total_time`, `v_daily_attendance`, `get_my_dashboard()` |
| `006_checklist_activity.sql` | Checklist Activity | Extends `log_activity` trigger to cover `task_checklist_items` |

See [docs/database.md](docs/database.md) for the full table reference and RLS policy summary.

## Design System

- **Font**: Montserrat for all body text; Instrument Serif italic for accent headings only
- **Colors**: Use CSS custom properties only вЂ” never raw hex values in components
  - `var(--navy)` вЂ” primary brand color (#10125A)
  - `var(--accent-brand)` вЂ” orange accent (#CE7E37)
  - Semantic status and urgency tokens (defined in global CSS)
- **Components**: shadcn/ui primitives live in `components/ui/`; `@base-ui/react` is used for headless primitives where shadcn/ui does not cover the use case
- **Theming**: `next-themes` with a `ThemeProvider` in `components/providers/`

## Out of Scope (v1)

The following features are intentionally excluded from the v1 release:

- OAuth / social login
- Real-time updates / WebSockets
- File uploads
- Multi-tenant organizations
- Mobile / Android app
- Comments and @mentions
- Recurring tasks UI
- Multi-assignee UI (schema supports it; UI defaults to single assignee)
- Role-based permissions system
- Manager views of team attendance

## Known Security Issues (Unfixed)

The following items were identified in a security audit. The seven issues found have been resolved; these five remain open:

- **No rate limiting** on auth endpoints (`/login`, `/register`, `/forgot-password`) вЂ” Supabase's built-in limits apply but no application-level throttle is in place
- **CSRF**: `SameSite=Lax` on auth cookies provides partial coverage; there is no explicit CSRF middleware
- **Session scope**: `signOut({ scope: "local" })` clears only the current browser session; other devices remain authenticated
- **Admin client guard**: protection against importing `lib/supabase/admin` in client components is enforced by an ESLint `no-restricted-imports` rule only вЂ” not a runtime guarantee
- **PostCSS CVE**: GHSA-qx2v-qp2m-jg93 is bundled in Next.js 16 вЂ” waiting for an upstream patch

## Contributing

Follow the conventions in [CLAUDE.md](CLAUDE.md):

- **File/folder names**: `kebab-case`
- **Variables and functions**: `camelCase`
- **Types, interfaces, and components**: `PascalCase`
- **Database column names**: `snake_case`
- **API responses**: always `{ data: ... }` on success and `{ error: { message, code? } }` on failure вЂ” use `ok()` and `ApiError.*` from `lib/api/response.ts`
- **Route handlers**: every protected route wraps its handler with `withAuth()` from `lib/api/handler.ts`
- **Validation**: run Zod `safeParse` before any database call; return `ApiError.badRequest()` on failure
- **Styling**: never use raw hex values вЂ” use CSS custom property tokens only (`var(--token-name)`)
- **IDs**: always `uuid`, generated by Postgres `gen_random_uuid()`
