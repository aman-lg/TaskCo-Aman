# Development Guide

## Local Setup

### Option A: Cloud Supabase project

1. Create a project at [supabase.com](https://supabase.com).
2. Go to **Project Settings → API** and copy the Project URL, anon key, and service role key.
3. Go to the **SQL Editor** and run `supabase/migrations/COMBINED_RUN_IN_DASHBOARD.sql` in full. This executes all migrations in the correct order (enums → tables → functions/triggers → RLS → views/RPC → checklist activity).
4. Copy `.env.example` to `.env.local` and fill in the values (see the Environment Variables section below).
5. `npm install && npm run dev`.

### Option B: Local Supabase CLI

```bash
# Install the Supabase CLI (if not already installed)
npm install -g supabase

# Start local Supabase stack (requires Docker)
supabase start

# This outputs local URLs and keys — use them in .env.local
# NEXT_PUBLIC_SUPABASE_URL=http://127.0.0.1:54321
# NEXT_PUBLIC_SUPABASE_ANON_KEY=<anon key from output>
# SUPABASE_SERVICE_ROLE_KEY=<service role key from output>

# Push all migrations to the local database
supabase db push

# Start the Next.js dev server
npm run dev
```

## Environment Variables

Defined in `.env.example`. Copy to `.env.local` and fill in values. Never commit `.env.local`.

| Variable | Client-side? | Description |
|---|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` | Yes | Supabase project REST URL |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Yes | Supabase anon (public) key |
| `SUPABASE_SERVICE_ROLE_KEY` | No | Service role key — bypasses RLS; server-only |
| `RESEND_API_KEY` | No | Resend transactional email API key |
| `RESEND_FROM_EMAIL` | No | Sender address for system emails |
| `NEXT_PUBLIC_APP_URL` | Yes | Canonical app URL (used for email links, redirects) |

`NEXT_PUBLIC_*` variables are bundled into the browser JS. Never add `NEXT_PUBLIC_` to `SUPABASE_SERVICE_ROLE_KEY` or any other secret.

## Running Migrations

### Cloud (Supabase dashboard)

Paste the contents of `supabase/migrations/COMBINED_RUN_IN_DASHBOARD.sql` into the Supabase SQL Editor and run it. For incremental updates, run individual migration files in numeric order.

### Local CLI

```bash
# Apply all pending migrations
supabase db push

# Apply a specific migration
supabase migration up --target 006_checklist_activity
```

Migration files must be created in `supabase/migrations/` with a numeric prefix to control run order (e.g., `007_next_feature.sql`).

## Generating TypeScript Types

After any schema change, regenerate `types/database.types.ts`:

```bash
# Cloud project
supabase gen types typescript --project-id <your-project-id> > types/database.types.ts

# Local Supabase stack
supabase gen types typescript --local > types/database.types.ts
```

Commit the updated `types/database.types.ts`. All Supabase client calls are typed against this file.

## Coding Conventions

From `CLAUDE.md`:

| Item | Convention |
|---|---|
| File and folder names | `kebab-case` |
| Variables and functions | `camelCase` |
| Types, interfaces, React components | `PascalCase` |
| Database column names | `snake_case` |
| IDs | `uuid`, generated by Postgres `gen_random_uuid()` |
| CSS values | CSS custom properties only (`var(--token-name)`); never raw hex |
| API response shape | `{ data: T }` on success; `{ error: { message, code? } }` on failure |

### Never do

- Never expose `SUPABASE_SERVICE_ROLE_KEY` to the client. Never import `lib/supabase/admin` in `"use client"` files (an ESLint `no-restricted-imports` rule enforces this).
- Never ship a table without RLS enabled.
- Never commit `.env*` files (only `.env.example` is allowed in the repository).
- Never use raw hex in components — only `var(--token-name)`.
- Never sum `task_time_entries` for "hours worked" — use `attendance_sessions`.
- Never build anything in the Out of Scope list: roles/permissions, manager attendance views, file uploads, recurrence UI, multi-tenant, multi-checklist per task, mobile app, comments/mentions, realtime.

## Adding a New API Route

1. **Create the route file** at the correct path under `app/api/`. Use kebab-case for folder names. Named export the HTTP verbs (`GET`, `POST`, `PATCH`, `DELETE`).

2. **Wrap with `withAuth()`** from `lib/api/handler.ts`. This verifies the session and provides `{ user, params }` to the handler.

3. **Define and apply a Zod schema** before any database call. Return `ApiError.badRequest()` on parse failure.

4. **Use `ok()` and `ApiError.*`** from `lib/api/response.ts` for all responses.

5. **Let RLS do authorization**. Do not duplicate permission checks in the application layer. A zero-row result from a guarded write indicates RLS blocked it — return `ApiError.forbidden()`.

### Template

```typescript
// app/api/widgets/route.ts
import { type NextRequest } from "next/server";
import { createClient } from "@/lib/supabase/server";
import { withAuth } from "@/lib/api/handler";
import { ok, ApiError } from "@/lib/api/response";
import { z } from "zod";

const createWidgetSchema = z.object({
  name: z.string().min(1).max(100),
});

export const POST = withAuth(async (req: NextRequest, { user }) => {
  const body = await req.json().catch(() => null);
  if (!body) return ApiError.badRequest("Request body is required");

  const parsed = createWidgetSchema.safeParse(body);
  if (!parsed.success) {
    return ApiError.badRequest(parsed.error.issues[0]?.message ?? "Invalid input");
  }

  const supabase = await createClient();
  // eslint-disable-next-line @typescript-eslint/no-explicit-any
  const db = supabase as any; // required if TypeScript infers never on narrow selects
  const { data, error } = await db
    .from("widgets")
    .insert({ ...parsed.data, owner_id: user.id })
    .select("id, name, owner_id")
    .single();

  if (error) { console.error("[widgets POST]", error); return ApiError.internal(); }
  return ok(data, 201);
});
```

6. **Write unit tests** in `tests/unit/api/` using Vitest. Mock `createClient` and `withAuth` — see existing tests in `tests/unit/api/` for the pattern.

## Adding a New Page

### Pattern: Server Component fetches → Client Component renders

1. **Create a query function** in `lib/queries/<resource>.ts`. Accept a `SupabaseClient<Database>` and return typed data. Throw on error.

```typescript
// lib/queries/widgets.ts
import type { SupabaseClient } from "@supabase/supabase-js";
import type { Database } from "@/types";

export async function getWidgets(supabase: SupabaseClient<Database>) {
  const { data, error } = await supabase
    .from("widgets")
    .select("id, name, created_at")
    .order("created_at", { ascending: false });
  if (error) throw new Error(error.message);
  return data ?? [];
}
```

2. **Create the page** as a Server Component in `app/(app)/<route>/page.tsx`. Call the query function and pass results as props.

```typescript
// app/(app)/widgets/page.tsx
import { createClient } from "@/lib/supabase/server";
import { getWidgets } from "@/lib/queries/widgets";
import { WidgetsClient } from "@/components/widgets/widgets-client";

export default async function WidgetsPage() {
  const supabase = await createClient();
  const widgets = await getWidgets(supabase);
  return <WidgetsClient widgets={widgets} />;
}
```

3. **Create the Client Component** in `components/<resource>/<resource>-client.tsx` marked `"use client"`. Handle interactivity (mutations, state) here. Call `/api/...` routes for writes.

```typescript
// components/widgets/widgets-client.tsx
"use client";
// ... form, fetch("/api/widgets", { method: "POST", body: JSON.stringify(...) })
```

4. **Add the route to the sidebar** in `components/layout/sidebar.tsx` if it needs navigation.

## Zod Validation Schemas

Schemas live in `lib/validations/`. Each file exports named schemas and their inferred TypeScript types:

| File | Schemas |
|---|---|
| `lib/validations/auth.ts` | `loginSchema`, `registerSchema`, `forgotPasswordSchema`, `resetPasswordSchema` |
| `lib/validations/projects.ts` | `createProjectSchema`, `updateProjectSchema` |
| `lib/validations/tasks.ts` | `createTaskSchema`, `updateTaskSchema`, `assignTaskSchema`, `createChecklistItemSchema`, `updateChecklistItemSchema` |

### `datetimeField` pre-processor

Both `projects.ts` and `tasks.ts` use a shared `datetimeField` pre-processor that:
- Converts an empty string or nullish value to `undefined` (field omitted)
- Converts `"YYYY-MM-DDTHH:mm"` (output of `<input type="datetime-local">`) to a full ISO 8601 string via `new Date().toISOString()`
- Passes through any already-valid ISO string unchanged

## Test Structure

```
tests/
├── unit/
│   ├── setup.ts              # Vitest global setup (jsdom, @testing-library/jest-dom)
│   ├── api/
│   │   ├── logout.test.ts    # POST /api/auth/logout
│   │   ├── projects.test.ts  # GET + POST /api/projects
│   │   └── tasks.test.ts     # GET + POST /api/tasks
│   └── validations/
│       ├── auth.test.ts      # loginSchema, registerSchema, etc.
│       ├── projects.test.ts  # createProjectSchema, updateProjectSchema
│       └── tasks.test.ts     # createTaskSchema, updateTaskSchema, checklist schemas
└── e2e/
    ├── auth.spec.ts          # Login, register, forgot-password flows
    └── logout.spec.ts        # Logout flow
```

### Running tests

```bash
# All unit tests (single run)
npm run test

# Unit tests in watch mode
npm run test:watch

# End-to-end tests
npm run test:e2e

# Playwright UI (interactive test runner)
npm run test:e2e:ui
```

Unit tests use Vitest with jsdom and `@testing-library/react`. End-to-end tests use Playwright against a running dev or preview server.
